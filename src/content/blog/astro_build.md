本篇文章将教学如何使用Astro部署静态博客网站，将资源托管在Github仓库中，并搭配使用CloudFlare的工人完成自动部署在公共互联网中。这也是本网站的构建历史。

## 关于 Astro

常见的SSG（Static Site Generation）技术有很多，例如 Hugo, Next.js, Nuxt.js 等等，本站使用Astro构建，因此以此为例。但该教程适用于所有SSG技术。

Astro 是一个专为构建快速、内容驱动型网站（如博客、营销网站、文档、作品集）而设计的现代化开源Web前端框架，与其他框架相比它减少了 JavaScript 的开销和复杂性。如果你需要一个加载速度快、具有良好 SEO 的网站，那么 Astro 是一个不错的选择。

[Astro 官网](https://astro.build/)

> 前提条件：  
> 需要安装 Node.js 运行时环境大于 v22.12.0 的更高双数版本。  
> 奇数版本是不被支持的，例如 v23。  
> 查询 Node.js 版本请使用 `node -v` 命令。  
> 下载 Node.js 请前往官网 https://nodejs.org/

Astro 提供了一键启动命令，允许你快速创建一个新项目。如果这是你第一次创建 Astro 项目，则该命令会首先配置好 Astro 环境：

```bash
npm create astro@latest
```

输入命令后，Astro 会向你显示向导对话，首先是需要你提供一个项目名称，也就是即将创建的文件夹名称。

你可以选择一路使用推荐配置完成向导对话，但 Astro 默认会给你提供一个模板项目。如果你希望完成从零开始搭建属于你自己的页面，则可以选择 Empty template 空白模板。

这里推荐勾选 Git 初始化以及 Install dependencies 选项，当然它们都是默认选项。

简单介绍一下安装后的项目结构：

- `src/`             ---> 源码目录
- `public/`          ---> 资源目录
- `package.json`     ---> 项目清单
- `astro.config.mjs` ---> Astro 配置文件
- `tsconfig.json`    ---> TypeScript 配置文件

你应该将图片等非代码资源存放在public/目录下，而将代码存放在src/目录下。

> 注意，`src/pages/`是一个特殊的目录，它规定了 Astro 最基本的页面路径。删除该目录会导致 Astro 无法正常工作！

常用的开发命令：

- `npm run dev`      ---> 启动服务器，你该前往默认 `http://localhost:4321` 网址上预览你的网站！你所看到的默认是 `index.astro` 的内容。
- `npm run build`    ---> 构建你的网站

Astro 支持热更新，这意味着你可以在不重启服务器的情况下，修改源码并立刻获得最新版本。

推荐使用 `localhost` 而不是 `127.0.0.1` 访问网站！

## 创建页面

在 `pages` 文件夹下方继续创建 `.astro` 文件，每一个文件都将被视为一个页面。

网页路由的方式则是从`pages/`为根，以文件名称为路由路径。例如`pages/myfirstpost`的网址将是 `localhost:4321/myfirstpost`，而如果在 `pages`下再创建一层文件夹`folder`，那么新的网址将是`localhost:4321/folder/myfirstpost`。

在默认生成的 `index.astro`中应该包含以下内容：

- 头元数据
- 正文内容

它们被 `---` 分隔，且头元数据的内容不会介入渲染页面，仅正文内容可以被正常渲染。

编写`Markdown`时使用标准 `#` 标题会自动生成隐式链接，以便你可以直接跳转到当前页面的某些部分。Astro 中的 `Markdown` 支持由`remark`提供，这是一个强大的解析和处理工具。因此，Astro并不一定强制页面以`.astro`格式存在，也直接支持`.md`格式，另外也支持`.mdx`，`.html`，`.js`等格式。（其中`.mdx`需要插件）

以下是一个普通的页面示例，上例为`.html`，下例则是`.astro`并使用`markdown`正文：

``````html
---
---
<html lang="en">
  <head>
    <title>我的主页</title>
  </head>
  <body>
    <h1>欢迎来到我的网站</h1>
  </body>
</html>
``````

``````markdown
---
title: 我的主页
---
# 标题

欢迎来到我的网站
``````

Astro 的路由策略是基于文件的，这意味着即使你的页面内容不在`pages/`目录下，也可以通过动态路由的方式生成多个页面。`src/pages/` 目录中的每个文件都会根据其文件路径成为网站上的一个端点。

要注意的是，Astro并没有集成路由配置，当`pages/`目录新增文件时，会自动生成一个路由。如果需要配置，请使用`build.format`。

## 项目结构

Astro和其他大多数SSG一样，使用一套约定俗成的文件夹布局来形成整个项目的架构。一个标准的Astro项目应该如以下所示：

- src/
- public/
- package.json
- astro.config.mjs
- tsconfig.json

其他常见的结构还有：

- src/components/
- src/layouts/
- src/styles/

在编写页面之前，应当具备组件设计思维，将页面中的元素拆分成多个可复用组件，并集中存放在`src/components/`中。当多个页面共享一套类似的框架结构时应该抽象成一份可复用布局，并集中存放在`src/layouts/`中。当多个组件多个元素共享同一个样式时，应当统一风格并写成可复用样式，并集中存放在`src/styles/`中。

这样做的好处是减少样板代码，大幅降低重复工作并提升效率，集中统一样式和风格，不至于出现多个文件布局不一致的尴尬。

> 本文对组件设计思维不做讨论。

## 云存储和版本控制

假如你还不了解什么是版本控制，极度推荐先去恶补一下相关知识。

**版本控制是任何长期性作业的根基，不会版本控制就无法开始任何项目，哪怕只是一个简易博客网站。**

> 下文默认你已知晓版本控制。

云存储的作用有二，其一是允许多设备多地方随时访问同一份文件，这样即使你外出或者不在平时的小屋中，也可以立即开始编辑你的内容。其次是对外公开一个稳定的访问点，毕竟总不能把家里的私人电脑给暴露到公网上吧。同时，云存储也是`CICD`（持续集成交付）的基础。

对于云存储有很多方案，如果不会自己搭建服务器，那么最好的办法就是使用现成的服务。在云存储领域中，`Github` 是最有名的代码托管服务之一，它对静态博客网站的部署也十分友好，甚至有内置的`Github Page`或者`Github Wiki`供你开箱使用。

**`Github`是一个免费，免登录的代码托管服务平台，不需要付费即可享受无限仓库的福利。对于下载者而言，无需任何账号即可访问和下载任意资源。`Github`并不是存储服务器也不是网盘服务，尽量不要将大量图片，音频，视频等非源码内容资源托管至此，否则有可能会被制裁。**

> 下文默认使用Github作为云存储服务，你可以使用其他服务。在Github开设仓库需要一个账户，创建账户的方式非常简单，这里不作赘述。

开设完仓库后将本地的静态博客项目推送至`Github`远端，云存储就完成了。如果你不知道如何推送项目至远端，最简单的方法是下载并使用 `Microsoft Visual Studio Code`，这是一个轻量的多功能编辑器，支持部分IDE功能和Git功能集成。

你只需要在`Visual Studio Code`授权登录你的`Github`账号即可，随后只需要在集成的Git功能中点击`推送`以及`同步`按钮即可。你可能需要填写`Git commit`信息，这是对一次提交的补充信息，可以任意填写。

**`Git`和`Github`是两个东西，请不要混为一谈。**

有了版本控制之后，每次你完成内容更新后都需要进行一次`推送`和`同步`。当内容被源源不断的推送至远端的`Github`时，它会自动帮你生成并维护各个版本的快照。这意味着即使你一不小心删除了某些重要内容且无法恢复的时候，只需要前往`Github`查看过往推送版本的快照即可。

使用`Git`命令甚至可以在多个平行版本之间进行自由切换，版本之间内容互相独立互不影响。

`Git`还贴心了提供了历史回退的功能，你可以自由的选择回退到某一个想要的版本时刻。

截至此处，如果愿意的话，`Github`提供了`Github pages`，可以将你的静态博客源码直接变成网站，并会直接附赠一个 `https://<你的账号名称>.github.io`的免费域名。**但是请注意，此功能在免费账户下仅针对公开仓库可用。**

如果即希望保持源码不公开，但又想要用SSG生成网站，那么你可以选择使用`Cloudflare Worker`。

## Cloudflare Worker

作为世界知名的大好人，`Cloudflare`也提供了一个近乎万能的免费套餐（~~至少对于静态博客网站部署而言~~），同时，`Cloudflare`的`Worker`功能可以连接到你现有的`Github`仓库，**不论其是否公开**。

将`Github`仓库设置为私有，这样除你之外的人就无法看见你的网站源码。然后使用`Cloudflare Worker`授权并连接到你的仓库，就可以执行自动部署了。

需要注意的是，**在授权`Cloudflare`连接`Github`仓库时，请将可授权范围控制在仅限特定仓库中**，即托管网站源码的仓库。

`Cloudflare Worker`自带了`CICD`功能，这意味着你不需要再在`Github`仓库中维护一份`Github Action`，只要你进行内容推送，`Cloudflare`就能自动收集并构建你的网站。

和`Github pages`类似的是，`Cloudflare`也会附赠一个免费的域名 `<你的项目>.workers.dev`，搭配上世界最快的`CDN`，你的博客网站将可在顷刻间全球访问。

## 域名

免费的域名固然好，但拥有一个特别的，可读性佳的，独属于你自己的域名是更好的事情。

**域名就像是互联网时代的身份证，且会越来越稀缺，如果有中意的域名推荐不要犹豫，要尽早购买。**

> 请选择专业的大型域名服务提供商，不要贪图免费域名服务。域名一旦丢失再想找回来的几率近乎为0。

`Cloudflare`同样提供域名服务，你可以在`cloudflare`上直接购买你所喜爱的域名，并将其直接绑定在你的静态博客网站上。这里推荐使用二级域名，例如`blog.xxx.com`，这样可以最大限度的利用单个域名进行多个不同任务。**子域名是不限数量的，可以大胆放心创建。**
